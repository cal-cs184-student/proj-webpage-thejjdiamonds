Part 2: Simulation via numerical integration

A Bounding Volume Hierarchy (BVH) acceleration structure is used to organize objects in a scene for the purpose of speeding up ray-object intersection tests. It starts by dividing the scene into a hierarchy of bounding boxes, and subdivides them recursively until each leaf node contains only a small number of primitives. When a ray is cast into the scene, it is tested against the bounding volumes at each level of the hierarchy, which enables the renderer to quickly eliminate large groups of objects that the ray does not intersect. This leads to a significant reduction in the number of intersection tests that need to be performed, resulting in faster rendering times.

Our BVH construction algorithm recursively creates each node in the BVH according to the max_leaf_size passed in and our splitting point heuristic. The process is as follows:

  • First, we compute the bounding box of the entire list of primitives passed in (from the iterators start to end) and initialize a new BVHNode node with that bounding box.
  • If the number of primitives in this bounding box is less than or equal to max_leaf_size, we simply set this new node‚Äôs start and end iterators to be the same as the ones passed in, and return node. This is our base case.
  • Otherwise, we must start the process of splitting the primitives into two parts and creating a BVHNode for the left and right children recursively.
    • Our splitting heuristic was to choose the longest axis of the current bounding box that all the primitives lie in, then split the primitives based on which half of this axis they lie on.
    • We made a lambda function that would take in a primitive and return true if its centroid lay to the left of the midpoint of the longest bounding box axis, and false otherwise.
    • We passed this lambda function into std::partition, which would sort the list in-place based on this function. It returns an iterator which we called mid that points to the part of the list where the primitives stop landing on one side of the axis and belong to the other.
    • Finally, we simply made the two recursive calls to BVHAccel::construct_bvh on the left and right members of node, passing in the iterators start and mid for the left child, and the iterators mid and end for the right child.
  • Our splitting scheme has a chance of choosing a split point such that all primitives lie on only one side of the split point. To avoid the infinite recursion of this case, we made an extra check through the iterators after std::partition is called to see if any part of the list has zero primitives. If we found that this was the case, we simply divided the remaining list of primitives in half, set the mid iterator to the middle of this list, and proceeded to the recursive calls.

Part 3: Handling collisions with other objects

To support cloth collision, we wrote an implementation that would support cloth collision with a sphere and cloth collision with a plane.

For sphere collision, we first check whether the point mass is in the sphere, which is done based on its distance from the center of the sphere. If the distance was less than the sphere‚Äôs radius, then the position of the point mass was located inside the sphere. From there, we calculated the tangent point (if we extended the path between the point mass‚Äôs position and the sphere‚Äôs origin to the sphere‚Äôs surface, the intersection with the sphere‚Äôs surface would be the tangent point). The tangent point was calculated by finding the direction between the point mass and origin and then multiplying it by the radius. We then added that result to the origin (effectively extending to the sphere‚Äôs surface). We then computed the correction vector, which was the difference between the tangent point and pm.last_position. Finally, pm.position was updated with pm.last_position + correction_vector * (1 - this->friction).

Below, we can see the images of the cloth‚Äôs resting position at varying values of ks. At ks = 5000, we see the default value of ks. The differences are more apparent when compared to ks = 500 and ks = 50000. At 500, the cloth over the sphere droops down a bit more (where it appears that the cloth is more weighted down than the default value of ks). At 50000, the cloth is more ‚Äúinflated‚Äù than the default value of ks, and appears to be more elevated (in other words, droops less over the sphere).

For plane collision, it‚Äôs a similar process to the sphere collision. In this case, we want to check if the point mass moved from one side of the plane to the other. In other words, we want to check if the point mass intersected with the plane based on pm.position and pm.last_position. For those variables, we found their difference with the point on the plane. Taking those differences, we found their dot product with the plane‚Äôs normal respectively to determine which side of the plane they were on - if the dot products had opposite signs, it implied that an intersection with the plane did occur. We then calculated the tangent point based on the position of the point mass, the plane‚Äôs normal and the dot product between the plane‚Äôs normal and pm.position - this->point (point on the plane). The correction_vector was based on the difference between the tangent point and pm.last_position with the addition of a surface offset to account for a slight displacement above the tangent point. Finally, pm.position was updated the same as was done in the sphere.

Part 4: Handling self-collisions

In our implementation of indirect lighting function, we first implemented the function sample_f that would help us take a sample for the function at_least_one_bounce_radiance. In this function, we considered the case of when max_ray_depth = 0 and when max_ray_depth = 1. In the former situation, we would return zero bounce illumination; in the latter case, we would return one bounce illumination. This was to account for the cases where we would only have direct illumination. So, specifically for indirect illumination, when max_ray_depth was greater than 1, we wanted to ensure that we always did at least 1 indirect bounce and that once that bounce was done, Russian Roulette would account for whether we would terminate on a certain bounce. To account for both cases, we had two checks in our code. For our first check: if the current ray‚Äôs depth was equal to the max ray depth, that implied that we haven‚Äôt had 1 indirect bounce yet (and this is assuming that max_ray_depth is greater than 1). For our second check, we set our termination probability to 0.35 (so the probability of the algorithm continuing would be 0.65), and we used the coin_flip function to determine if Russian Roulette would terminate the process. Assuming that the process wasn‚Äôt terminated, we casted our ray and decremented the ray‚Äôs depth and then checked its intersection with our bvh. If it intersected, then we recursively called on at_least_one_bounce_radiance and the recursive result was factored into L_out. More specifically, the recursive result was multiplied by our sample from sample_f and the derived cosine factor and then divided by the pdf and the probability of continuation.

Part 5: Shaders

Adaptive sampling improves the efficiency of rendering. Before, we would take a fixed number of samples from each pixel for our Monte Carlo integration algorithm, regardless of how much detail is present in the scene. This can result in images that are noisy or blurry, particularly in areas with high contrast or fine details. Adaptive sampling works by dynamically adjusting the number of samples taken in each pixel based on the complexity of the scene. In areas with high detail or contrast, more samples are taken to capture the fine details and reduce noise. In areas with less detail or lower contrast, fewer samples are taken to reduce rendering time without sacrificing image quality. The algorithm gradually increases the number of samples until a desired level of image quality is achieved. This approach significantly minimized rendering time and improved the overall quality of the image.

In our implementation of adaptive sampling, we followed the specifications and computed the illuminance of estimated radiance using the Vector3D::illum() function. We also stored two running sums, s1 and s2, where s1 holds the illuminances and s2 holds the squares of the illuminances. After every samplesPerBatch iterations, we perform a convergence check by calculating the mean and variance using s1 and s2 to determine if the condition I <= maxTolerance * ùúá has been satisfied. If it has, we break out of the loop and update the sampleBuffer and sampleCountBuffer accordingly.
